# 2026-05-25
- Added cooldown before duplication is allowed to each new block.
  - Results in a slower explosion of blocks.
- Expanded play space to width height of page.
- Added collision grid optimisation.
- Removed the map of colour to block list and just have one array of blocks.
- Flipped the collision win condition, now the colour with fewer total blocks wins.
- Added a cross wall in the middle.
- Increased the block speed.
- Added win screen.
- Added collision grid size check before duplicating blocks.
  - If collision grid cell has as many blocks it can fit by area then don't add anymore.
  - This should help the case where the visible volume of a colour is not proportional to its count due to too much overlapping going on.
  - When this happens it can have win conditions that don't look like the winning colour has actually won.
  - Dynamically set the win block count based on page area now that there's a cap based on area filled.
    - Without this blocks may never reach the arbitrary win count.
    - Added a cap to what the win count can be in case the user has an enormous screen that would eat all their CPU to fill up.

# 2026-05-20
- Idea from friend.
  - A variant of these videos: https://www.youtube.com/@simulando2d
  - Start with two blocks of red, yellow, green, blue.
  - If two blocks of the same colour collide they double to four blocks.
  - If two different colours collide, whichever has fewer total blocks will be deleted.
  - If they have the same count then they just bounce off each other.
- Implemented blocks and walls.
  - Rendering.
  - Movement.
  - Collision detection.
  - Trail.
  - Collision bouncing.
  - Different colour collision handling.
  - Max block detection and reset.
- Turns out after just a few same colour collisions the number of blocks explodes very very quickly. Probably exponentially.
